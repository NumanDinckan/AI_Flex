# Scenario Comparison Table

Source file: `rq3/scenario_comparison_table_2025.csv`

The table compares the RQ2 flex-only cases and the RQ3 flex-plus-BESS cases against the original 2025 utilisation profile.

## How The Table Was Created

The table is generated by `make_scenario_comparison_table()` in `src/rq3_figure2.py`. It is written automatically when RQ3 is generated through:

```bash
python src/intermediate_report_exports.py --rq rq3 --year 2025 --output-dir .
```

The function uses the full-year half-hour `year_df` after the complete pipeline has run:

1. load and filter the UKPN data-centre utilisation CSV
2. attach the UK electricity price CSV if available
3. apply RQ2 flexibility and create `load_flex_10` and `load_flex_25`
4. run RQ3 BESS dispatch and create the BESS-adjusted load columns
5. compute the comparison metrics from the full-year columns

The table is not calculated from the annual mean-day or mean 48-hour figure. It is calculated from the full-year half-hour series.

## Input Columns

Original baseline:

- `utilisation`

Flex-only scenarios:

- `load_flex_10`
- `load_flex_25`

Flex plus BESS scenarios:

- `load_10_flex_4h_batt`
- `load_10_flex_8h_batt`
- `load_25_flex_4h_batt`
- `load_25_flex_8h_batt`

Price input:

- `uk_price`

In the committed 2025 run, every load timestamp has a matched UK price value.

## Formulas

Original annual peak:

```text
original_peak = max(utilisation)
```

Scenario residual peak:

```text
residual_peak = max(scenario_load)
```

Annual peak reduction:

```text
annual_peak_reduction_vs_original_percent =
    (original_peak - residual_peak) / original_peak * 100
```

Original price-weighted cost proxy:

```text
original_cost_proxy = sum(utilisation * uk_price * dt_hours)
```

Scenario price-weighted cost proxy:

```text
scenario_cost_proxy = sum(scenario_load * uk_price * dt_hours)
```

Price-weighted cost reduction:

```text
price_weighted_cost_reduction_vs_original_percent =
    (original_cost_proxy - scenario_cost_proxy) / original_cost_proxy * 100
```

For the current run, `dt_hours = 0.5` because the data is half-hourly.

## Why These Columns

The table is limited to six columns so it stays usable in the final report and presentation:

- `scenario`
- `flex_case`
- `bess_duration`
- `residual_peak_utilisation`
- `annual_peak_reduction_vs_original_percent`
- `price_weighted_cost_reduction_vs_original_percent`

The chosen metrics are consistent with common evaluation practice:

- Demand-response and data-centre flexibility studies often report peak avoidance or coincident-peak reduction.
- Price-aware load-shifting studies often report electricity cost or bill savings.
- Behind-the-meter battery studies often report demand-peak reduction and price/tariff-driven dispatch effects.

The residual peak column is kept because it anchors the percentage reduction in the remaining load level.

## Interpretation

The column choices follow common data-centre demand-response and storage evaluation practice: peak reduction captures grid and capacity relevance, while electricity-price-weighted cost reduction captures whether the shifted or battery-adjusted profile moves consumption away from expensive hours.

The current table shows a clear division between the two modelling layers. Flex-only dispatch reduces the annual peak from `0.3684` to `0.3616` in both flex cases, a `1.8355%` reduction. The `25%` flex-only case does not improve the annual peak beyond the `10%` case, and after price is excluded from the flex selector it also has only a `0.0102%` price-weighted cost proxy reduction. This supports interpreting RQ2 as a constrained operational load-shifting layer, not as the main peak-reduction mechanism.

BESS supplies the larger peak effect. The strongest annual-peak case is `25% Flex + 8h BESS`, with residual peak `0.3404` and `7.6045%` peak reduction versus original. The strongest price-weighted cost proxy case is `25% Flex + 4h BESS`, with `0.9013%` reduction. This difference is expected because the `8h` batteries are energy-matched but lower-power variants, while the `4h` batteries retain higher instantaneous power for price-timed dispatch within the peak cap.

The cost column is a price-weighted proxy:

`sum(load * UK electricity price * dt_hours)`

Because the load data is in utilisation ratios, the values should be interpreted as relative scenario comparisons, not absolute electricity bills. The metric does not include demand charges, fixed charges, taxes, network tariffs, battery degradation, or investment cost.

## Validation Checks

The current table was checked against an independent recomputation from the full-year series:

- 2025 half-hour rows used: `17520`
- UK price signal used: `True`
- original peak: `0.3683996547544206`

The table values match the recomputed values after rounding to four decimals.

Flex-only energy is conserved at annual scale. BESS cases can have higher total grid energy than the flex-only cases because battery charging/discharging includes round-trip losses, but the price-weighted cost proxy can still fall if charging is shifted toward lower-price periods.

## Caveats

- The cost metric is a relative proxy, not a monetary bill.
- The load unit is utilisation, not MW.
- The table does not include demand charges, fixed charges, network tariffs, taxes, BESS degradation, or BESS investment cost.
- The values depend on the selected flexibility method and BESS dispatch method. If either method changes, regenerate the table.
